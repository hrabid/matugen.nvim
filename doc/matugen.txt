1. Introduction					*matugen-intro*

matugen.nvim is a colorscheme for Neovim that dynamically loads colors
generated by the `matugen` CLI tool. It reads VSCode-compatible JSON files
to provide a seamless experience across editors.

The plugin uses a semantic palette system. Color values from the JSON file
are mapped to named keys (e.g., `surface`, `primary`, `on_surface`) and then
applied to Neovim highlight groups via modular templates.

Hex colors with alpha (9-character #RRGGBBAA), shorthand (4-character #RGB),
or 5-character (#RGBA) formats are automatically normalized to 7-character
#RRGGBB.

==============================================================================
2. Configuration				*matugen-config*

To configure the plugin, pass options to the setup function.

Example configuration with lazy.nvim:
>lua
<{
  "Senal-D-A-Gunaratna/matugen.nvim",
  lazy = false,
  priority = 1000,
  opts = {
    load_theme = true,

    -- Path to the JSON file generated by matugen
    palette_path = "~/.config/matugen/themes/nvim-colors.json",
  },
},
<
`palette_path` must point to a `.json` file. If the file is missing,
unreadable, or contains invalid JSON, the plugin falls back to a built-in
dark theme and notifies you. Missing color keys in the palette are filled
from the fallback palette.

Settings:
  `palette_path`    (string)  Path to the matugen JSON palette file.
                              Default: `""` (empty — uses fallback).

  `load_theme`      (boolean) Whether to load the theme on startup.
                              Default: `true`.

==============================================================================
3. Commands					*matugen-commands*

*:MatugenReload*
	Clears the template cache, re-reads all template files from disk,
	and reloads the theme from the currently configured `palette_path`.
	Use this after modifying template files or the palette file.

==============================================================================
4. Signal Support				*matugen-signals*

The plugin listens for the `SIGUSR1` signal to trigger a theme reload.
This allows you to automate theme updates via scripts (e.g., when your
wallpaper changes). Signal-based reloads run asynchronously and use
the in-memory template cache to prevent blocking the editor loop.

Example usage:
>bash
pkill -SIGUSR1 nvim
<

5. Lua API						*matugen-api*

The plugin exposes several functions from the `matugen` module for
programmatic use in your config or autocmds.

`setup(opts)`						*matugen-api-setup*

	Initializes the plugin with the given options. When `load_theme` is
	true (the default), loads the theme asynchronously on startup.

	Parameters: ~
		`opts`		(table, optional) Options table:
				- `palette_path`	(string) Path to the matugen
					JSON palette file. Must end in `.json`.
					Default: `""`.
				- `load_theme`		(boolean) Whether to load
					the theme on startup. Default: true.

	Example:
	>lua
	require("matugen").setup({
	    palette_path = "~/.config/matugen/themes/nvim-colors.json",
	    load_theme = true,
	})
	<

`load(on_done, force_sync)`				*matugen-api-load*

	Reads the palette file from the configured `palette_path`, parses
	it, maps colors to the semantic palette, and applies all highlight
	templates. Runs asynchronously (non-blocking) by default.

	When the palette path ends in `.json`, the file is read via libuv
	(async) or `io.open` (sync). A generation counter prevents stale
	async reads from overwriting newer palette data.

	If the file is missing or fails to parse, a built-in dark fallback
	palette is used and a warning is issued.

	Color values are validated: every value must be a hex color
	starting with `#`. If validation fails, the fallback palette is
	used instead and a one-time warning is shown.

	Parameters: ~
		`on_done`	(function, optional) Callback invoked after all
				highlights are applied. When provided, the
				parsed palette data is cached so that
				subsequent calls to `load_theme` can
				re-apply highlights without re-reading the
				file.

		`force_sync`	(boolean, optional) If true, performs file I/O
				synchronously. Used internally by the
				colorscheme entry point.

	When called without arguments, reads the palette file asynchronously
	via libuv and defers highlight application via `vim.schedule`.

`load_theme(force_sync)`				*matugen-api-load_theme*

	Convenience wrapper around |matugen-api-load| that calls `load`
	and then applies the `colorscheme matugen` command. This is used
	internally by `setup` when `load_theme` is true.

	Parameters: ~
		`force_sync`	(boolean, optional) Defaults to true. Pass
				false for an async non-blocking load.

`reload_templates()`					*matugen-api-reload_templates*

	Clears the in-memory template cache. The next call to `load` or
	`load_theme` will re-read all template files from disk. Note that
	this does not automatically re-apply highlights; call `load_theme`
	afterwards to apply the newly loaded templates.

Templates are loaded from `lua/matugen/templates/`, pinned to the plugin's
own directory to prevent rogue plugins from injecting files via runtimepath.
Each file must return a function accepting `(c, hl)` where `c` is the color
palette table and `hl` is a wrapper around `vim.api.nvim_set_hl`.

`_status`						*matugen-api-_status*
	Internal string field reflecting the last load status (e.g.,
	"Loaded successfully"). Not written to `vim.g`.

`_template_count`					*matugen-api-_template_count*
	Internal integer field reflecting the number of loaded templates.

`_last_reload`						*matugen-api-_last_reload*
	Internal os.time() value of the last successful reload.

`_palette_path`						*matugen-api-_palette_path*
	Internal string of the last loaded palette path (tilde-expanded).

6. Health Check					*matugen-health*

You can verify the status and configuration of matugen.nvim by running:
>vim
:checkhealth matugen
<
This command checks:
  - Neovim version is 0.10 or higher.
  - matugen CLI is installed (optional — informational).
  - Palette file existence, readability, and JSON parse validity.
  - All color values in the palette are valid hex codes.
  - Loaded palette path, last reload timestamp, active template count,
    and overall load status.
